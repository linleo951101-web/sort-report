## 1. 氣泡排序法（Bubble Sort）

### 原理簡述
氣泡排序會重複走訪陣列，比較相鄰兩個元素，若順序錯誤就交換。每一輪都會把目前最大的元素「浮」到陣列尾端，因此稱為氣泡排序。

### 複雜度分析
- 時間複雜度（最佳）: O(n)（已排序且有提早停止機制）
- 時間複雜度（平均）: O(n^2)
- 時間複雜度（最差）: O(n^2)
- 空間複雜度: O(1)
- 穩定性: 穩定

### 模擬實作內容
- 逐步比較相鄰元素並交換。
- 每輪完成後，最後一個元素固定在正確位置。
- 實作中可加入「本輪是否交換」旗標，若未交換可提前結束。

## 2. 選擇排序法（Selection Sort）

### 原理簡述
每一輪從未排序區間中選出最小值，放到未排序區間的起始位置。重複執行直到整個陣列完成排序。

### 複雜度分析
- 時間複雜度（最佳）: O(n^2)
- 時間複雜度（平均）: O(n^2)
- 時間複雜度（最差）: O(n^2)
- 空間複雜度: O(1)
- 穩定性: 不穩定（一般寫法）

### 模擬實作內容
- 以雙層迴圈掃描資料。
- 外層固定目前位置，內層找最小值索引。
- 每輪只交換一次，交換次數少但比較次數固定多。

---

## 3. 插入排序法（Insertion Sort）

### 原理簡述
把陣列視為「已排序區」與「未排序區」。每次取出未排序區第一個元素，插入已排序區的正確位置。

### 複雜度分析
- 時間複雜度（最佳）: O(n)（原本接近排序完成）
- 時間複雜度（平均）: O(n^2)
- 時間複雜度（最差）: O(n^2)
- 空間複雜度: O(1)
- 穩定性: 穩定

### 模擬實作內容
- 使用一個 key 暫存待插入元素。
- 往左比較並搬移較大元素。
- 找到正確位置後把 key 放入。

---

## 4. 合併排序法（Merge Sort）

### 原理簡述
採用分治法，把陣列遞迴切分成左右兩半，分別排序後再合併成有序陣列。

### 複雜度分析
- 時間複雜度（最佳）: O(n log n)
- 時間複雜度（平均）: O(n log n)
- 時間複雜度（最差）: O(n log n)
- 空間複雜度: O(n)
- 穩定性: 穩定

### 模擬實作內容
- 以遞迴方式拆分區間。
- 合併時使用兩個指標比較左右子陣列。
- 將較小值依序放入暫存陣列，最後回寫。

---

## 5. 快速排序法（Quick Sort）

### 原理簡述
同樣採分治法，先選定一個 pivot（基準值），透過分割（partition）把小於 pivot 的元素放左邊、大於 pivot 的放右邊，再對左右區間遞迴排序。

### 複雜度分析
- 時間複雜度（最佳）: O(n log n)
- 時間複雜度（平均）: O(n log n)
- 時間複雜度（最差）: O(n^2)（分割極度不平均）
- 空間複雜度: O(log n)（遞迴堆疊平均）
- 穩定性: 不穩定（一般原地寫法）

### 模擬實作內容
- 選擇 pivot（可用尾端元素或中位策略）。
- 進行 partition，回傳 pivot 最終位置。
- 對左右子區間繼續遞迴直到區間長度 <= 1。

## 6. 綜合比較

| 排序法 | 最佳時間 | 平均時間 | 最差時間 | 空間複雜度 | 穩定性 | 適用情境 |
|---|---|---|---|---|---|---|
| 氣泡排序 | O(n) | O(n^2) | O(n^2) | O(1) | 穩定 | 教學示範、資料量小 |
| 選擇排序 | O(n^2) | O(n^2) | O(n^2) | O(1) | 不穩定 | 交換成本高但資料量小 |
| 插入排序 | O(n) | O(n^2) | O(n^2) | O(1) | 穩定 | 幾乎排序完成的小型資料 |
| 合併排序 | O(n log n) | O(n log n) | O(n log n) | O(n) | 穩定 | 需要穩定排序與可預期效能 |
| 快速排序 | O(n log n) | O(n log n) | O(n^2) | O(log n) | 不穩定 | 一般情況效能佳、實務常用 |

### 比較結論
- 若資料量小或教學用途，插入排序與氣泡排序較直觀。
- 若重視穩定且效能要穩定，合併排序較合適。
- 若追求平均效能與實務速度，快速排序常是首選，但需注意最差情況。

## 7. 學習心得

這次整理各種排序法後，我更能理解「演算法設計」與「資料特性」的關係。表面上每個方法都在做排序，但核心思維差很多：

- 氣泡、選擇、插入偏向逐步修正，邏輯好理解，但在大量資料下效率不足。
- 合併、快速屬於分治策略，雖然實作較複雜，但在時間效率上明顯更好。

我在模擬過程中最大的收穫是：同一個問題可以有多種解法，應該依照資料規模、是否需要穩定性、記憶體限制等條件選擇排序法，而不是只看程式能不能跑出結果。未來在寫程式時，我會先評估需求與複雜度，再決定要用哪種演算法。
